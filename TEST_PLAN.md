# 时差信 测试计划

> 本计划覆盖核心业务流程的端到端测试用例，每次发版前需全量回归。
> 版本号与 `pages/settings/settings.js` 中的 `data.version` 保持同步。

---

## 模块一：用户注册与登录

### 场景 1.1：新用户首次登录
- [ ] 步骤：启动小程序，使用微信授权登录
- [ ] 预期：用户记录写入 `users` 集合，openid 正确，未配对状态
- [ ] 验证：首页显示"建立连接"而非"书写"

### 场景 1.2：老用户重新打开小程序
- [ ] 步骤：已登录用户再次打开小程序
- [ ] 预期：自动通过 openid 恢复登录状态，显示配对/未配对历史状态
- [ ] 验证：Settings 页显示正确的昵称和送信时间

### 场景 1.3：登录信息持久化
- [ ] 步骤：登录后退出小程序，再重新打开
- [ ] 预期：无需重新授权，自动恢复登录态

## 模块二：邀请码匹配

### 场景 2.1：生成邀请码
- [ ] 步骤：未配对用户点击"建立连接"
- [ ] 预期：生成唯一邀请码，页面展示二维码和数字码
- [ ] 验证：邀请码存入 `invites` 集合，状态为 pending，有过期时间

### 场景 2.2：使用邀请码配对（正常）
- [ ] 步骤：另一用户输入有效邀请码并确认
- [ ] 预期：双方 `isPaired = true`，`partnerOpenid` 互相写入，invite 状态变为 accepted
- [ ] 验证：双方首页均显示"书写"按钮，Settings 显示"已连接"

### 场景 2.3：邀请码过期
- [ ] 步骤：使用已过期的邀请码（> 24h）
- [ ] 预期：云函数返回错误，提示邀请码已失效
- [ ] 验证：无法完成配对

### 场景 2.4：重复配对
- [ ] 步骤：已配对用户再次生成邀请码
- [ ] 预期：无法对自己发起配对，云函数返回错误

### 场景 2.5：取消/换账号配对
- [ ] 步骤：用户 A 发出邀请，用户 B 接受前用户 A 退出登录，换 C 账号接受
- [ ] 预期：用户 A 的邀请码只被 B 使用，C 无法使用（对应关系正确）

## 模块三：通信设置

### 场景 3.1：设置送信时间（自己的解锁时间）
- [ ] 步骤：已配对用户在 Settings 页点击送信时间，输入 21:00
- [ ] 预期：时间存入自己的 user 记录，首页提示更新
- [ ] 验证：我发的信按 21:00 解锁（对方在 21:00 后才能看到）

### 场景 3.2：送信时间独立性
- [ ] 步骤：用户 A 设送信时间 21:00，用户 B 设送信时间 09:00
- [ ] 预期：A 发给 B 的信按 A 的 21:00 解锁；B 发给 A 的信按 B 的 09:00 解锁
- [ ] 验证：两人各自收到的 pending 时间提示不同，各读各的

### 场景 3.3：开启来信通知
- [ ] 步骤：已配对用户在 Settings 页点击"来信通知"，在微信弹窗中点击"允许"
- [ ] 预期：云函数 `updateNotifySubscribe` 被调用，`subscribed = true` 存入 DB
- [ ] 验证：UI 即时变为"已开启"；退出重进后仍显示"已开启"

### 场景 3.4：关闭来信通知
- [ ] 步骤：已开启通知的用户再次点击，选择拒绝
- [ ] 预期：`subscribed = false` 存入 DB，UI 即时更新为"未开启"

### 场景 3.5：取消来信通知弹窗
- [ ] 步骤：点击来信通知后，在微信弹窗中点"取消"
- [ ] 预期：不做任何状态变更，不弹 toast

### 场景 3.6：设置昵称
- [ ] 步骤：在 Settings 页点击昵称，输入"小明"，确认
- [ ] 预期：昵称存入 DB，Settings 页立即显示"小明"
- [ ] 验证：退出重进后仍显示"小明"；对方时光轴中显示我的昵称

## 模块四：写信与收信

### 场景 4.1：正常发送一封信
- [ ] 步骤：已配对用户点击"书写"，输入内容后点击"封缄"
- [ ] 预期：信件存入 `letters` 集合，`unlockAt = now + 1day @unlockTime`，`isRead = false`
- [ ] 验证：发信后按钮变为"今日已书"；对方 pending 提示出现

### 场景 4.2：每日一封信限制
- [ ] 步骤：当天已发送一封信，再次点击"书写"
- [ ] 预期：按钮为"今日已书"（disabled 状态），点击无反应或提示今日已写过

### 场景 4.3：次日重置写信权限
- [ ] 步骤：跨天后（本地日期变更）再次打开
- [ ] 预期：可再次发送，"今日已书"按钮恢复为可点击状态

### 场景 4.4：信件按时解锁
- [ ] 步骤：设置送信时间 22:00，在 22:00 前发一封信
- [ ] 预期：22:00 前对方看不到（unlockAt 未到）；22:00 后刷新页面可见
- [ ] 验证：已收到页面显示解锁时间；未解锁时无信件内容

### 场景 4.5：收信后标记已读
- [ ] 步骤：收到新信件后点击查看
- [ ] 预期：弹出信件内容（modal）；云函数 `markLetterRead` 被调用
- [ ] 验证：信件状态变为已读；首页未读红点消失

### 场景 4.6：未读红点数字正确
- [ ] 步骤：收到 N 封未读信件
- [ ] 预期：首页"已收到"入口显示未读数字 N

### 场景 4.7：时光轴双向显示
- [ ] 步骤：双方各发一封信
- [ ] 预期：时光轴显示 4 条记录（每人各 2 条：自己发1 + 收到1），带正确昵称

### 场景 4.8：连续发信不积压
- [ ] 步骤：用户 A 连续两天发信，用户 B 在第二天才上线
- [ ] 预期：A 的两封信分别在不同时间解锁；B 上线后看到正确的 pending 提示

## 模块五：信件收藏

### 场景 5.1：收藏一封收到的信件
- [ ] 步骤：打开一封收到的信件，点击「收藏」按钮
- [ ] 预期：按钮变为「已收藏」（seal-color 背景），toast 提示「已收藏」
- [ ] 验证：返回收信列表，该信件左边框变为金色

### 场景 5.2：取消收藏
- [ ] 步骤：打开已收藏信件，点击「已收藏」按钮
- [ ] 预期：按钮恢复为「收藏」，toast 提示「已取消收藏」
- [ ] 验证：返回列表，金色边框消失

### 场景 5.3：收藏一封寄出的信件
- [ ] 步骤：在「已寄出」页面打开自己发出的信件，点击「收藏」
- [ ] 预期：收藏成功，返回送信列表可见金色左边框

### 场景 5.4：收信页「仅收藏」筛选
- [ ] 步骤：收藏若干信件后，在收信页点击「仅收藏」
- [ ] 预期：列表只显示已收藏的信件，计数更新
- [ ] 验证：再次点击关闭筛选，恢复全量列表

### 场景 5.5：送信页「仅收藏」筛选
- [ ] 步骤：同 5.4，在「已寄出」页面操作
- [ ] 预期：筛选正常，开关状态正确

### 场景 5.6：时光轴「仅收藏」筛选
- [ ] 步骤：在时光轴页面点击「仅收藏」
- [ ] 预期：只显示已收藏的时间线条目，词云区域隐藏
- [ ] 验证：关闭筛选后恢复完整时间线

### 场景 5.7：收藏状态跨页面同步
- [ ] 步骤：在信件详情收藏后，分别检查收信页、时光轴的金色边框
- [ ] 预期：所有页面 onShow 刷新后一致显示收藏标记

### 场景 5.8：双人独立收藏
- [ ] 步骤：A 和 B 收藏同一封信（A 发给 B 的信）
- [ ] 预期：各自独立，A 取消不影响 B 的收藏状态
- [ ] 验证：双方分别查看收藏筛选结果互不干扰

### 场景 5.9：无收藏时的空状态
- [ ] 步骤：在没有任何收藏的情况下打开「仅收藏」筛选
- [ ] 预期：显示空状态「尚未收藏信件 / 在信件详情中点击收藏」

## 测试环境说明

- 测试需在**微信开发者工具真机调试**或**体验版真机测试**，云函数在模拟器环境下行为可能与真机不同
- 两人测试建议各用一台真机，轮流操作
- 发版前在「微信公众平台后台 → 管理版本」确认云函数列表中所有函数均为最新部署状态
